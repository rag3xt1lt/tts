СЛАЙД 1 — Название и цель
- Кейс: «Чёрный бармен» (AntiHack 2026)
- Цель этапа: с нуля сделать клон API, который совпадает с референсом по ответам и скрытой логике
- Референс: https://bar.antihype.lol
- Ограничения: JSON, Bearer-токен, поддержка X-Time, порт 8000


СЛАЙД 2 — Что было в задании (входные данные)
- Документация описывает только базовые эндпоинты
- Остальное нужно исследовать экспериментами: запросы → ответы → сравнение поведения
- Оценка зависит от точности клона и найденных «скрытых» механик/секретов


СЛАЙД 3 — Выбранный стек и почему
- Язык/фреймворк: Python + FastAPI
- Сервер: Uvicorn
- Почему: быстрый старт, удобно отдавать JSON и быстро менять поведение по диффам
- Запуск ориентирован на Windows (cmd/PowerShell), Docker — опционально


СЛАЙД 4 — Структура проекта (что создано)
- Папка проекта: D:\black-barmen
- main.py — реализация API и логики
- requirements.txt — зависимости
- README.md — команды установки/запуска/проверки
- Dockerfile — вариант контейнерного запуска


СЛАЙД 5 — Авторизация и модель данных
- POST /register создаёт аккаунт и выдаёт token
- Все остальные эндпоинты требуют Authorization: Bearer <token>
- При отсутствии/невалидности — HTTP 401
- Хранение состояния аккаунта (in-memory):
  - balance (старт 100)
  - orders (история)
  - total_orders, unique_drinks
- POST /reset сбрасывает аккаунт к начальному состоянию


СЛАЙД 6 — Реализованные базовые эндпоинты
- POST /register
- POST /reset
- GET /menu (поддержка X-Time: HH:MM)
- POST /order (ошибки: unknown_drink, insufficient_funds)
- POST /mix (ошибки: unknown_recipe, insufficient_funds)
- GET /balance
- POST /tip (ошибка: insufficient_funds)
- GET /history
- GET /profile


СЛАЙД 7 — Почему нужен был инструмент сравнения
- Документация неполная → нельзя “угадать” поведение
- Нужно быстро находить отличия и править только то, что реально не совпадает
- Поэтому добавлен автоматический сравнитель референс vs локальный сервер


СЛАЙД 8 — probe.py (сравнение с референсом)
- Скрипт регистрируется на референсе и локально
- Прогоняет одинаковый сценарий запросов (menu/order/mix/tip/history/profile + reset)
- Сравнивает status_code и JSON-тела ответов
- Печатает точные диффы полей/значений
- После правок позволяет проверить прогресс одной командой


СЛАЙД 9 — Что нашли по диффам и как исправили
- Меню референса оказалось больше → расширили список напитков и цены
- Формат id у референса: BAR-12345 → привели генерацию к такому виду
- X-Time влияет на поведение → добавили правила под наблюдения референса:
  - для /menu mood_level всегда normal
  - невалидный X-Time (например, 99:99) возвращает отдельное «ночное меню»
- /tip при insufficient_funds не содержит лишних полей → привели формат ошибки
- /profile: поля и ранги подогнаны под наблюдаемое поведение на сценарии


СЛАЙД 10 — Технические нюансы Windows (что учли)
- venv и py_compile могут не печатать вывод при успехе (это нормально)
- Сервер подтверждённо запускается и отвечает на 127.0.0.1:8000
- Отладка баланса и истории показала, что повторный запрос = повторное списание (проверяли через /history)


СЛАЙД 11 — Результат этапа (чёткий итог)
- Локальный сервер слушает порт 8000
- Базовый набор эндпоинтов реализован и совпадает по форме ответов
- probe.py показывает: “OK: local matches reference for this probe scenario.”
- Это означает: на текущем проверочном сценарии локальное поведение соответствует референсу


СЛАЙД 12 — Следующий шаг: поиск скрытых эндпоинтов
- Добавлен discover.py: аккуратно прогоняет wordlist путей по референсу
- Печатает всё, что отличается от обычного 404 (кандидаты на скрытые эндпоинты/фичи)
- Дальше план: найденные хиты повторить в локальном API и добавить в сценарий сравнения


СЛАЙД 12.1 — Этап “доведение до идентичного поведения” (как делали)
- Расширили probe.py: добавили edge-кейсы и скрытые проверки
  - /secret с токеном и без токена
  - X-Time: 23:59 и невалидный 99:99
  - /mix с разным порядком ингредиентов
  - /tip с отрицательным значением (валидация)
- После каждой правки прогоняли probe.py до результата “OK: local matches reference…”


СЛАЙД 12.2 — Что нашли как “секрет”
- discover.py упёрся в rate limit референса (429 rate_limit, retry_after)
- Реальный найденный скрытый путь: GET /secret
  - без авторизации: 401 и особый JSON в поле detail
  - с авторизацией: 200 и {status:"error", error:"not_found"}
- Это поведение повторили в main.py (и закрепили проверкой в probe.py)


СЛАЙД 12.3 — Стабильность тестов
- Из-за сети/ограничений референса возможны таймауты TLS-handshake
- В probe.py добавлены ретраи и понятные подсказки (timeout/retries/insecure)


СЛАЙД 13 — Команды (для демонстрации на показе)
1) Запуск сервера (локально):
   - python -m uvicorn main:app --host 127.0.0.1 --port 8000
2) Проверка совпадения на сценарии:
   - py probe.py --ref https://bar.antihype.lol --local http://127.0.0.1:8000 --timeout 60 --retries 4
3) Разведка скрытых эндпоинтов:
   - py discover.py --ref https://bar.antihype.lol --sleep-ms 1500 --max-requests 30
